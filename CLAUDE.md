始终使用中文回复

1. 技术栈统一：项目统一使用 Next.js + TypeScript + React + Tailwind CSS v4 + shadcn/ui + SQLite，禁止随意引入其他库和技术。

2. 目录结构规范：遵循 Next.js 官方推荐的项目结构，页面放在 `app/` 或 `pages/` 目录，组件放在 `components/` 目录，数据库相关操作放在 `db/` 目录。

3. 组件开发规范：
   - 所有组件使用 TypeScript 书写，严格类型校验。
   - UI 组件优先使用 shadcn/ui 提供的组件，如需自定义组件，遵循开源组件风格并使用 Tailwind CSS 进行样式控制。
   - 组件命名使用大驼峰风格（PascalCase），hooks 统一放在 `hooks/` 目录，命名使用小驼峰（camelCase）。

4. 样式规范：
   - 仅使用 Tailwind CSS v4 进行样式开发，禁止使用内联 style 和其他 CSS 方案。
   - 禁止覆盖 Tailwind 核心类，如需全局定制统一放在 `globals.css` 内。
   - 优先采用 shadcn/ui 推荐的设计系统风格。

5. 路由和页面规范：
   - 使用 Next.js 内置路由，页面组件命名与路径一致。
   - 路由跳转统一使用 `next/navigation` 的钩子或 Link 组件。

6. 数据库操作：
   - 所有数据库操作均通过 PostgreSQL + Drizzle ORM，数据库操作代码集中管理，避免在页面或组件中直接写 SQL。
   - 使用 Drizzle ORM 或自己封装数据库层，注重 SQL 注入安全。

7. 数据类型与接口：
   - 严格使用 TypeScript 定义所有类型与接口，包括前后端数据交互的数据结构。
   - 提前定义好 API 请求与响应的接口类型，减少 bug 风险。

8. 代码风格与规范：
   - 使用 Prettier 和 ESLint 保持代码风格统一，提交代码前确保无 lint 错误和格式问题。
   - 统一使用函数式组件并采用 hooks 编写逻辑，无特别需求不使用 class 组件。

9. 版本管理与分支规范：
   - 代码统一托管到 Git，采用 Git Flow 或 feature 分支开发模式，分支命名规范如 `feature/xxx`，`bugfix/xxx`。
   - 合并代码需 PR 审核，并要求描述清楚修改内容。

10. 自动化与测试：
    - 优先为核心业务逻辑和复杂组件编写单元测试。
    - 大功能、重要更新需在开发和测试环境自测通过后再合并上线。

11. 文档与协作：
    - 重要模块、组件及接口需有注释和说明文档（如 README、API 文档）。
    - 新同事或贡献者需优先查阅现有文档，发现文档不全及时完善补充。

12. 安全与性能：
    - 注意接口及数据库安全，防止 SQL 注入和 XSS 漏洞。
    - 合理拆分页面与组件，避免不必要的重渲染，提高性能。
    - 静态资源（如图片）优先使用 CDN，页面图片使用 Next.js 的 `Image` 组件优化加载。

13. 持续集成与部署：
    - 尽量采用 CI/CD 工具自动化测试与部署，确保主干分支始终可用。
    - 部署前通过构建工具压缩优化代码体积。

14. 持续改进：
    - 定期团队会议，回顾技术选型和实现方法，不断优化开发流程和项目质量。
